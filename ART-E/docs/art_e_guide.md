# ART-E コード解説

ART-E の `art_e.py` がどのように動くかを解説します。Agentic RL（エージェント強化学習）の仕組みを、コードの構造に沿って理解できるようにしています。

## 全体アーキテクチャ

`art_e.py` は大きく5つのフェーズで構成されています。


## Rollout: エージェントが1問を解く

`rollout()` は、エージェントに1つの質問を与え、ツールを使ってメールを検索し、回答を生成するまでの1エピソードです。これが ART-E の中核です。

```
                    ┌─────────────────────────────────┐
                    │  ユーザーの質問                    │
                    │  「Xについてのメールを探して」      │
                    └──────────────┬──────────────────┘
                                   ▼
┌──────────────────────────────────────────────────────────────┐
│                  エージェントループ（最大6ターン）              │
│                                                              │
│   ┌──────────────┐                                           │
│   │  LLM 推論     │◀─────────────────────────────────┐      │
│   │  (Qwen 3 14B) │                                  │      │
│   └──────┬───────┘                                   │      │
│          ▼                                           │      │
│   ┌──────────────┐    Yes    ┌──────────────────┐    │      │
│   │ ツール呼び出し │─────────▶│ ツール実行         │────┘      │
│   │ あり？        │          │                  │            │
│   └──────┬───────┘          │ search_inbox()   │            │
│     No   │                  │ read_email()     │            │
│          ▼                  │ return_final_    │            │
│    テキスト応答で終了         │   answer()       │            │
│                             └────────┬─────────┘            │
│                                      ▼                      │
│                             ┌──────────────────┐            │
│                             │ final_answer?     │            │
│                             │  Yes → 評価して終了│            │
│                             │  No  → 次のターン  │            │
│                             └──────────────────┘            │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### コードの対応箇所

```python
@weave.op
async def rollout(model: art.Model, email_scenario: EmailScenario) -> ProjectTrajectory:
```

- `@weave.op`: この関数の実行を Weave でトレース記録する
- 戻り値の `ProjectTrajectory` にメッセージ履歴、メトリクス、最終回答が入る

### 3つのツール

エージェントは以下の3つのツールを使い分けて回答を導きます。

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   search_inbox(keywords)                                │
│   ┌─────────────────────────────────────────────┐       │
│   │ キーワードでメールボックスを全文検索           │       │
│   │ → 件名・送信者・日付のリストを返す            │       │
│   └─────────────────────────────────────────────┘       │
│                         ▼                               │
│   read_email(message_id)                                │
│   ┌─────────────────────────────────────────────┐       │
│   │ 特定のメールIDから本文を取得                   │       │
│   │ → メールの全内容を返す                        │       │
│   └─────────────────────────────────────────────┘       │
│                         ▼                               │
│   return_final_answer(answer, reference_message_ids)    │
│   ┌─────────────────────────────────────────────┐       │
│   │ 回答テキストと根拠メールIDを提出              │       │
│   │ → RULER 評価 → スコア記録                    │       │
│   └─────────────────────────────────────────────┘       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

ツールは `langchain_core` の `convert_to_openai_tool` で OpenAI の Tool 形式に変換され、モデルの推論リクエストに渡されます。

## RULER 評価: LLM が正誤を判定する

`judge_correctness()` は、エージェントの回答が正しいかを **別の LLM（GPT-4.1）** に判定させます。

```
┌───────────────────────────────────────────────────┐
│              judge_correctness()                   │
│                                                   │
│  入力:                                            │
│  ┌─────────────────────────────────────────────┐  │
│  │ Question:  「Xについてのメールを探して」      │  │
│  │ Reference: 「2001年のメールにXXと記載あり」    │  │
│  │ AI Answer: 「XXXという内容でした」            │  │
│  └─────────────────────┬───────────────────────┘  │
│                        ▼                          │
│  ┌─────────────────────────────────────────────┐  │
│  │         ジャッジモデル (GPT-4.1)              │  │
│  │                                             │  │
│  │  「参照回答と比較して、                       │  │
│  │   AI回答が正しいか判定してください」           │  │
│  └─────────────────────┬───────────────────────┘  │
│                        ▼                          │
│  出力:                                            │
│  ┌─────────────────────────────────────────────┐  │
│  │ reasoning: "回答は参照と一致している"          │  │
│  │ accept:    true                              │  │
│  └─────────────────────────────────────────────┘  │
│                                                   │
└───────────────────────────────────────────────────┘
```

### コードの対応箇所

```python
@weave.op
@retry(stop=stop_after_attempt(3))
async def judge_correctness(scenario, answer) -> CorrectnessJudgeResponse:
```

- `@weave.op`: Weave でトレース
- `@retry`: LLM の応答が不安定な場合に最大3回リトライ
- `CorrectnessJudgeResponse` は `reasoning`（理由）と `accept`（正誤）を持つ Pydantic モデル

## GRPO + RULER: 強化学習の仕組み

ART-E の強化学習は **GRPO**（Group Relative Policy Optimization）と **RULER**（相対スコアリング）の組み合わせで行われます。

```
┌───────────────────────────────────────────────────────────────┐
│                    1ステップの学習                              │
│                                                               │
│  シナリオ A ──┐                                               │
│              ├──▶ TrajectoryGroup                             │
│              │    ┌─────────────────────────────────────┐     │
│  rollout ×4  │    │  回答1: 「XXです」     correct=1    │     │
│              │    │  回答2: 「わかりません」 correct=0    │     │
│              │    │  回答3: 「XXです」     correct=1    │     │
│              │    │  回答4: 「YYです」     correct=0    │     │
│              │    └─────────────────┬───────────────────┘     │
│              │                     ▼                          │
│  シナリオ B ──┘    ┌─────────────────────────────────────┐     │
│                   │         RULER スコアリング            │     │
│                   │                                     │     │
│                   │  LLM ジャッジが4つの回答を相互比較     │     │
│                   │  → 相対スコア [0.8, 0.1, 0.8, 0.3]  │     │
│                   └─────────────────┬───────────────────┘     │
│                                     ▼                         │
│                   ┌─────────────────────────────────────┐     │
│                   │            GRPO 更新                 │     │
│                   │                                     │     │
│                   │  高スコアの回答を出す方向に重みを更新   │     │
│                   │  絶対スコアは不要、相対的な順位で学習   │     │
│                   └─────────────────────────────────────┘     │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### なぜ GRPO + RULER なのか

| 手法 | 特徴 |
|------|------|
| **GRPO** | 同じ質問に対する複数回答の「相対的な良さ」で学習する。絶対スコア（0〜100点）を設計する必要がない |
| **RULER** | LLM ジャッジが複数回答を相互比較してスコアを付ける。人手のラベル付け不要 |

従来の RL では報酬関数の設計が難しかったが、GRPO + RULER の組み合わせにより「LLM が LLM を評価する」形で自動化できます。

## トレーニングループ: train()

`train()` は全体の流れを制御するメイン関数です。

```
train()
│
├── Weave 初期化
│   weave.init("entity/project")
│
├── モデル初期化
│   ┌──────────────────────────────────────────┐
│   │  TrainableModel 作成                     │
│   │       ▼                                 │
│   │  Backend 選択                            │
│   │  ├─ ServerlessBackend（デフォルト）       │
│   │  │   → CoreWeave GPU で実行              │
│   │  └─ LocalBackend（--local 指定時）       │
│   │       → ローカル GPU で実行              │
│   │       ▼                                 │
│   │  model.register(backend)                 │
│   └──────────────────────────────────────────┘
│
├── シナリオ読み込み
│   ├─ train: 20件（Enron メールデータセット）
│   └─ val:   5件
│
└── ステップループ（max_steps=5）
    │
    ├── Step 1 ─────────────────────────────────
    │   │
    │   ├── Rollout 収集
    │   │   2 groups × 4 rollouts = 8回実行（並列）
    │   │
    │   ├── RULER スコアリング
    │   │   各グループを ruler_score_group() で評価
    │   │
    │   └── GRPO でモデル更新
    │       model.train(judged_groups)
    │
    ├── Step 2 ─────────────────────────────────
    │   └── (同上)
    │
    ├── ...
    │
    └── Step 5 ─────────────────────────────────
        ├── (同上)
        └── バリデーション実行（5ステップごと）
            val シナリオ 5件で評価 → model.log()
```

### コードの対応箇所

```python
async def train(config: Config):
    set_config(config)
    weave.init(config.model.project, ...)
    model = await initialize_model(config)

    # シナリオ読み込み
    training_scenarios = load_scenarios(split="train", ...)
    validation_scenarios = load_scenarios(split="test", ...)

    for batch in training_iterator:
        # 1. 各シナリオに対して複数回 rollout
        train_groups = [
            art.TrajectoryGroup(
                rollout(model, scenario) for _ in range(rollouts_per_group)
            )
            for scenario in batch.items
        ]

        # 2. 並列でロールアウト収集
        finished = await art.gather_trajectory_groups(train_groups)

        # 3. RULER スコアリング
        judged = await ruler_score_group(group, judge_model)

        # 4. GRPO でモデル更新
        await model.train(judged, config=art.TrainConfig(learning_rate=1e-5))
```

## データモデル

### CorrectnessJudgeResponse

RULER の LLM ジャッジが返す構造化レスポンス。

```python
class CorrectnessJudgeResponse(BaseModel):
    reasoning: str   # 「参照回答と一致しているため正解と判断」
    accept: bool     # True = 正解 / False = 不正解
```

### ProjectTrajectory

ART の `Trajectory` を拡張。1回のエージェント実行の全履歴を保持する。

```python
class ProjectTrajectory(art.Trajectory):
    final_answer: FinalAnswer | None = None
```

親クラス `Trajectory` が持つフィールド:

```
ProjectTrajectory
├── messages_and_choices   # メッセージ履歴（system → user → assistant → tool → ...）
├── reward                 # 報酬値
├── metrics                # {"correct": 1.0} など
├── tools                  # OpenAI Tool 形式のツール定義
├── metadata               # {"scenario_id": "...", "step": 3}
└── final_answer           # FinalAnswer(answer="...", source_ids=[...])
```

### EmailScenario

`rollout()` に渡すためのラッパー。

```python
class EmailScenario(BaseModel):
    step: int          # 現在のトレーニングステップ番号
    scenario: Scenario # 質問、期待される回答、メールボックス情報
```

## 設定の優先順位

設定は3段階で上書きされます。

```
  ┌──────────────────────────────────────┐
  │  config.py のデフォルト値              │   最も低い優先度
  │  max_steps=5, groups_per_step=2, ... │
  └──────────────────┬───────────────────┘
                     ▼  上書き
  ┌──────────────────────────────────────┐
  │  環境変数（.env ファイル）             │
  │  WANDB_ENTITY, WANDB_PROJECT         │
  └──────────────────┬───────────────────┘
                     ▼  上書き
  ┌──────────────────────────────────────┐
  │  コマンドライン引数                    │   最も高い優先度
  │  --max-steps 500 --project my-proj   │
  └──────────────────────────────────────┘
```

`config.model.project` は `"entity/project"` 形式で統一管理されます。ART の `TrainableModel` に渡すときだけ `entity` と `project_name` に分割されます。
